# 产品更新日志 — Lulu's Daily Mic

## v1.0 — 初始版本发布

### 迭代基础信息
- **更新时间**：2026-05-18
- **版本号**：v1.0
- **迭代类型**：新增

### 本次改动核心功能

1. **口语脚本生成**：支持从 YouTube/Bilibili/网页/主题四种输入源生成约2分钟英文口播脚本。脚本包含固定打卡开篇（Hi guys, it's Lulu. Day X of my daily English practice...）和固定结尾（Thank you for listening, see you tomorrow.），附带完整中文对照翻译。脚本风格严格约束为日常简单词汇+短句+客观转述视角，适配小红书短视频口播节奏。

2. **AI PM 学习材料推送**：支持早晚两次（10:00/22:00）自动策展推送 AI 产品经理学习资源，每次3-5条精选内容，附带中文推荐描述和分类标签。

3. **Web UI 网页端**：基于 Streamlit 的三页面界面，温暖学术风格设计。脚本生成页（输入→生成→双语展示→下载）、脚本历史浏览页（搜索→展开→下载/删除）、学习材料浏览页（最新推送横幅→材料列表→历史记录）。

4. **CLI 命令行**：基于 Rich 的交互式终端界面，支持命令模式和自然语言输入，适配定时任务触发。

5. **macOS 原生通知**：定时推送触发后弹出系统通知，引导用户打开网页查看完整内容。

### 本次使用 AI 技术

- **LLM**：DeepSeek-chat（deepseek-chat），通过 OpenAI 兼容接口调用，JSON mode 强制结构化输出
- **Prompt Engineering**：双重约束策略——固定开篇/结尾在 prompt 中重复三次强调（正文+格式说明+结尾提醒），词汇限制使用「违反即失败」的强语气
- **内容抓取**：youtube-transcript-api（免费免 API key）、Bilibili 公开 API、trafilatura 网页正文提取
- **调度机制**：Claude Code CronCreate 定时触发 + macOS osascript 原生通知
- **存储方案**：纯 JSON 文件存储，零数据库依赖，索引文件 + 实体文件分离
- **前端框架**：Streamlit 1.28 + 自定义 CSS（温暖学术风：奶油色底、Newsreader 衬线字体、琥珀色点缀）

### 开发落地难点

1. **Prompt 约束力不足风险**：LLM 天生有「创造性输出」倾向，容易改写固定开篇结尾、使用更复杂的词汇和句式。解决方案是在 prompt 中用多层强调+惩罚性语言强化约束，并在生成后做自动检测。

2. **内容抓取覆盖率有限**：YouTube 不是所有视频都有字幕，Bilibili 字幕覆盖面更低。技术上采用了四级回退机制（英文字幕→中文字幕翻译→任意可用字幕→标题/简介），确保即使字幕缺失也能生成脚本。

3. **推送触达问题**：纯网页端无法主动触达用户——用户不打开网页就看不到推送。解决方案是引入 macOS 原生通知（osascript display notification）作为桥接层：CronCreate 触发脚本生成 → 保存数据 → 发系统通知（带生动文案引导） → 用户打开网页查看。

4. **产品定位与英语水平匹配**：市面工具要么面向初学者要么面向高级学者，精准匹配「雅思6.5 + AI PM职业场景」是一个空白地带。产品设计上选择以内容题材（AI PM相关）和语言难度（高频2000词+短句）作为差异化定位。

### 解决方案与取舍

**做的选择**：
- ✅ 用 DeepSeek 而非 Claude/GPT：成本优先，日常高频调用场景下性价比是首要考虑
- ✅ 用纯 JSON 文件存储而非 SQLite：零依赖、可手动编辑、面试可直接展示数据结构
- ✅ 用 Streamlit 而非 React/Next.js：开发速度优先，Streamlit 现有成熟模板可复用
- ✅ 用 macOS 通知而非 Telegram Bot：v1.0 先跑通核心流程，减少外部平台依赖
- ✅ 用免费字幕 API 而非 YouTube Data API：免除 API key 管理和费用控制

**放弃的方案**：
- ❌ 不做语音合成（TTS）集成：v1.0 聚焦脚本生成，语音朗读留到后续迭代
- ❌ 不做发音评分：涉及复杂的语音识别和评分算法，过度工程化
- ❌ 不做手机推送：先验证 macOS 桌面端使用场景，手机端后续加 Telegram Bot
- ❌ 不做用户系统/登录：单人使用场景，无需多用户隔离

### 产品复盘思考

本次迭代的核心收获是**「推送触达」问题的解决方式**。起初想直接做 Telegram Bot 解决手机推送，但后来发现对于「坐在电脑前学习」这个场景，macOS 桌面通知已经足够。这个取舍体现了做 AI PM 产品的一个重要原则：**先定义清楚使用场景，再选择技术方案，而不是反过来**。

另一个思考是关于 Prompt 设计中的「惩罚式约束」——用严厉语气（FAILURE IF VIOLATED、COPY EXACTLY）来对抗 LLM 的创造性。这是一种 Prompt Engineering 中的「对抗设计」思路，在需要严格格式输出的场景下有效，但也要注意不触发模型的抵抗行为。后续可以探索用 few-shot 示例替代惩罚式语言。

产品架构上刻意保持了高内聚低耦合，每个 Skill 独立运作，后续加 Telegram Bot 或语音功能只需加新模块，不影响现有流程。

## v1.0.1 — 网页文本颜色 bug 修复

### 迭代基础信息
- **更新时间**：2026-05-18
- **版本号**：v1.0.1
- **迭代类型**：修复

### 本次改动核心功能
修复 Web UI 在系统暗色模式下输入文字不可见的 bug：自定义 CSS 仅设置了浅色背景（奶油色/暖白色），但未覆盖 Streamlit 原生的文本颜色。当用户 macOS 处于暗色模式时，Streamlit 自动应用浅色文本，导致输入框、下拉框、展开面板等组件出现「白字白底」问题，用户完全看不清输入内容。

### 本次使用 AI 技术
- **Streamlit 主题配置**：在 `.streamlit/config.toml` 中新增 `[theme]` 配置节，显式声明 `base = "light"` 强制浅色主题，阻断系统暗色模式的自动跟随
- **CSS 全面覆盖**：为 `.stApp`、`.stTextInput`、`.stSelectbox`、`.stMarkdown`、`.stCaption`、`.stExpander` 等关键组件补充 `color` 属性，设置浏览器 autofill 样式覆盖，确保全部 UI 文字使用深色墨水色

### 开发落地难点
Streamlit 的主题系统默认跟随系统设定，而其 CSS 类名和 DOM 结构并非公开 API，需要通过浏览器 DevTools 逐个调试找到准确的 CSS 选择器。不同组件（输入框、下拉框、展开面板）使用的是不同的 BaseWeb 组件树，需要分别定位。

### 解决方案与取舍
- ✅ 采用「双保险」策略：config.toml 层（防止主题跟随系统）+ CSS 层（精确覆盖剩余遗漏组件）
- ✅ 在 `.stApp` 上设置全局 `color` 作为兜底，而非逐个覆盖所有可能的组件
- ❌ 不做暗色模式适配：产品定位为桌面白天学习场景，不需要暗色模式支持

### 产品复盘思考
这是一个典型的「自定义样式覆盖不完全」问题，在 Streamlit 这类高度封装的前端框架中尤其常见。框架默认行为的假设（light theme）和用户系统环境（dark mode）不一致时，就会出现这类视觉 bug。教训是：使用封装框架做自定义 UI 时，必须显式声明基础主题（base theme），再在此基础上做 CSS 覆盖，不能依赖隐式默认值。

## v1.0.2 — Quick Topics 报错修复 + YouTube 内容抓取增强

### 迭代基础信息
- **更新时间**：2026-05-19
- **版本号**：v1.0.2
- **迭代类型**：修复 + 优化

### 本次改动核心功能

1. **Quick Topics 点击报错修复**：修复点击快捷主题按钮时的 `StreamlitAPIException: st.session_state.script_input cannot be modified after the widget...` 错误。根因是 Streamlit 控件执行顺序问题 — text_input 控件先于 quick topics 按钮渲染，按钮尝试修改已被控件「占用」的 session_state key。

2. **YouTube 内容抓取失败不再胡乱生成**：当 YouTube 字幕抓取失败时，之前会将空壳 fallback 内容送入 LLM，导致生成完全无关的内容。现在改为直接向用户展示明确错误信息（含失败原因），阻断无效生成。

3. **YouTube URL 解析增强**：新增支持 `youtube.com/shorts/`、`youtube.com/live/`、`youtube.com/v/` 等 URL 格式。

4. **YouTube 字幕回退策略优化**：重构字幕获取优先级 — 手动英文字幕 → 自动生成英文 → 任意英文 → 中文翻译英文 → 任意语言翻译英文，每级用精准的 `NoTranscriptFound` 异常判断而非笼统 `except Exception`。

### 本次使用 AI 技术
- **Streamlit on_click 回调模式**：利用 `st.button(on_click=callback, args=(...))` 在 text_input 控件渲染前写入 session_state，规避 Streamlit 的控件所有权检查
- **youtube-transcript-api 细粒度异常处理**：引入 `NoTranscriptFound`、`TranscriptsDisabled`、`VideoUnavailable` 等特定异常类，替代笼统的 `except Exception`，实现精准的降级回退

### 开发落地难点

1. **Streamlit session_state 所有权机制**：Streamlit 要求每个 widget key 只能由对应 widget 写入。quick topics 按钮在 text_input 之后渲染，导致跨控件写入冲突。解决方案是将 quick topics 提前到 text_input 之前，利用 `on_click` 回调在控件创建前完成写入。

2. **YouTube 字幕 API 的不稳定性**：`youtube-transcript-api` 实际使用的是 YouTube 的内部 API，时有被限流或变更的风险。解决思路是建立完整的优先级回退链，确保即使某个层级失败也有备选方案，并在所有层级都失败时给出有意义的用户提示而非静默降级。

### 解决方案与取舍
- ✅ 将 quick topics 移至输入框上方：技术上解决了 session_state 冲突，UI 上快捷入口更醒目
- ✅ 使用 `on_click` 回调而非直接修改 session_state：符合 Streamlit 推荐模式
- ✅ 阻断 fallback 内容进入 LLM：宁可失败可见，也不生成垃圾内容
- ❌ 不做 YouTube oEmbed/Data API v3 集成：需 API key，与项目「零外部依赖」原则冲突

### 产品复盘思考
这次修复揭示了一个重要的产品设计原则：**「优雅降级」不等于「静默失败」**。原代码在内容抓取失败时做了 fallback，但 fallback 的质量不足以支撑 LLM 生成有意义的脚本，结果就是「看起来没报错，但产出的是垃圾」。正确的做法是：当核心输入质量不达标时，应该明确告知用户并阻止后续流程，而不是硬着头皮继续跑完 pipeline。

另一个收获是关于 Streamlit 的 session_state 模型理解。Streamlit 的 widget key 体系本质上是一个「每个 key 有且只有一个 owner widget」的约束，这和 React 的受控组件理念有相似之处。理解这个模型后，调整控件顺序和使用回调函数就成为了自然的解决方案。

## v1.0.3 — YouTube 字幕 API 兼容修复 + 学习材料质量增强

### 迭代基础信息
- **更新时间**：2026-05-19
- **版本号**：v1.0.3
- **迭代类型**：修复 + 优化

### 本次改动核心功能

1. **YouTube 字幕 API 兼容修复**：`youtube-transcript-api` 新版本中 `list_transcripts` 类方法已移除，改为实例方法 `list()`。修复了 `AttributeError: type object 'YouTubeTranscriptApi' has no attribute 'list_transcripts'` 错误，YouTube 链接恢复正常识别和字幕抓取。

2. **学习材料 URL 可访问性验证**：MaterialSkill 新增 `_check_url()` 方法，对 LLM 生成的每个链接做 HEAD/GET 双重验证。HEAD 失败时自动降级为 GET stream 验证，确保推送给用户的链接都是真实可访问的。

3. **精准 5 篇推送 + 重试机制**：LLM 生成 8 篇候选 → URL 验证过滤 → 取前 5 篇有效链接。不足 5 篇时自动重试（最多 3 次），确保每次推送数量准确且质量可靠。

4. **material_curator.txt 提示词重构**：新增 approved sources 白名单（Anthropic/OpenAI/Google/Meta 官方文档、LangChain/HuggingFace/GitHub 开发平台、The Verge/TechCrunch 等 AI 媒体、Lenny's/Pragmatic Engineer 等产品社区），约束 LLM 只用稳定平台的真实 URL，杜绝幻觉链接。

5. **Quick Topics UI 优化**：快捷主题按钮从全宽改为 pill 胶囊样式（圆角边框、hover 琥珀色高亮），Generate 按钮单独定制主色样式。

### 本次使用 AI 技术
- **youtube-transcript-api 新版本 API 迁移**：类方法 `list_transcripts()` → 实例方法 `api.list()`，TranscriptList 过滤 API 保持不变
- **URL 验证双保险**：HEAD 请求优先（轻量），失败后回退 GET stream（兼容不支持 HEAD 的服务器）
- **生成-验证循环模式**：LLM 生成候选 → 程序化验证 → 不达标自动重试，将 LLM 的不可靠性和程序的可靠性结合

### 开发落地难点

1. **youtube-transcript-api 版本漂移**：该库不在 `requirements.txt` 中锁定版本，pip 安装时自动获取最新版，导致 API 签名变化后直接崩溃。根因是没有锁版本号。后续需在 `requirements.txt` 中锁定 `youtube-transcript-api>=0.x,<1.0` 或指定具体版本。

2. **LLM 幻觉 URL 问题**：之前 material_curator 生成的 URL 有相当比例是 LLM 编造的（格式正确但页面不存在）。解决思路是「源头约束 + 事后验证」双管齐下：prompt 层用 approved sources 白名单限制生成范围，代码层用 HTTP 请求验证每个链接。这种「LLM + 程序化验证」的混合模式是 AI 产品中处理幻觉的通用范式。

### 解决方案与取舍
- ✅ 使用 `YouTubeTranscriptApi().list(video_id)` 实例方法，保持 5 级回退逻辑不变
- ✅ 新增 URL 验证 + 重试机制：牺牲少量生成时间（+5-10秒），换取 100% 链接可访问
- ✅ 用 approved sources 白名单替代「URL 必须真实」的模糊约束：给 LLM 具体边界比抽象要求更有效
- ❌ 不做 youtube-transcript-api 版本锁定：当前问题已修复，版本锁定留到后续统一处理

### 产品复盘思考
这次 YouTube 修复再次验证了一个教训：**外部依赖的 API 变更是最常见的静默炸弹**。`youtube-transcript-api` 作为一个非官方库，API 稳定性没有保证。更稳健的做法是在 CI 中加入依赖 API 兼容性检查，或在 `requirements.txt` 中锁定版本。但对于个人项目，过度工程化的版本管理不值得 — 快速发现、快速修复的节奏更适合 Vibe Coding 阶段。

MaterialSkill 的 URL 验证改造体现了一个 AI PM 产品设计的核心原则：**不要期望 LLM 能做所有事，把「确定性」留给代码，把「创造性」留给 LLM**。URL 是否存在是确定性判断，用代码验证；资源选题和描述是创造性工作，交给 LLM。这个边界划分让产品质量从「有时候链接打不开」提升到「每个链接都可访问」。

## v1.0.4 — Tab 导航从 HTML href 重构为 st.pills 图标+文字融合式标签

### 迭代基础信息
- **更新时间**：2026-05-19
- **版本号**：v1.0.4
- **迭代类型**：优化

### 本次改动核心功能
将三页签导航从自定义 HTML `<a href="?tab=...">` 链接重构为 Streamlit 原生 `st.pills` 组件。旧方案每次切换触发浏览器完整页面刷新（白屏闪烁），新方案利用 `session_state` 实现零页面刷新的毫秒级切换。每个 tab 块内图标（emoji）+ 文字融为一体，无独立 icon 行、无勾选圆圈，视觉上干净统一。

### 本次使用 AI 技术
- **st.pills（Streamlit 1.36+）替代 HTML href 导航**：原生块状选择器，`selection_mode="single"` + `session_state` 做 tab 状态管理，天然无 radio 指示器
- **CSS 自定义 st.pills 样式**：匹配品牌温暖学术风格（圆角顶部边框 + 琥珀色选中态 + 去阴影）

### 开发落地难点
旧方案使用 URL query params（`?tab=speak`）+ `target="_self"` 触发浏览器整页刷新。这是 Streamlit 早期常见的多页签实现方式，但体验很差。中间尝试过 `st.radio` 方案，但 radio 带有勾选圆圈指示器且 icon 与文字分离为两个区域，不符合设计要求。最终采用 `st.pills`，图标和文字天然融合在同一个块内，无任何 extra UI 元素。

### 解决方案与取舍
- ✅ 用 st.pills + session_state 替代 URL params：切换零黑屏，图标文字融为一体
- ✅ 保持三页签视觉效果和品牌调性一致
- ❌ 不再支持通过 URL 参数直接跳转到指定 tab：接受这个 trade-off

### 产品复盘思考
同样的切换需求，经历了 HTML href（全页刷新）→ st.radio（有勾选圆点，icon 分离）→ st.pills（图标文字融合，无多余 UI）三次迭代。选组件时不仅要看功能，还要看它自带的默认 UI 是否符合设计预期 — st.pills 的默认形态恰巧就是我们要的「图标+文字融合式块状按钮」，零额外改造。
